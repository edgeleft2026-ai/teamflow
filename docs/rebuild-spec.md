# TeamFlow 重建说明书

## 1. 文档目的

这份文档用于在项目推倒重来时，重新收敛目标、边界、关键流程和验收标准。

目标不是修旧系统，而是重新搭建一个更小、更稳、更容易维护的 TeamFlow。


## 2. 产品目标

TeamFlow 的核心目标只有一件事：

让 AI 成为一个基于飞书协作的项目管理助手，能够围绕项目的创建、沟通、跟踪、提醒和汇报形成闭环。

一句话版本：

`用户在飞书里发起项目 -> 系统创建项目协作空间 -> 后续自动同步状态、提醒风险、汇报进展`


## 3. 第一性原理

重建时必须坚持以下原则：

1. 先把主链路做通，再做智能化增强。
2. 先保证可观测、可定位、可回滚，再追求自动化深度。
3. 飞书交互先用最稳定的方式，避免一上来依赖复杂卡片表单。
4. 动作执行和对话交互解耦。
5. 所有自动动作必须有明确输入、明确输出、明确失败原因。
6. 不追求“大而全”，只保留能支撑真实项目协作的最小集合。


## 4. MVP 目标

### 4.1 必须实现

1. 飞书中可以触发项目创建。
2. 项目创建后，系统能在本地落库。
3. 项目创建后，系统能自动完成飞书协作空间初始化。
4. 后续可以围绕该项目发送通知、推送日报或周报、做任务提醒。
5. 出问题时能够快速知道卡在哪一步。

### 4.2 明确不在第一阶段做

1. 复杂多步卡片表单。
2. 过度抽象的策略系统。
3. 自动演化和自我优化。
4. 太多并行的消息入口。
5. 花哨但不稳定的交互组件。


## 5. 用户视角的核心流程

### 5.1 项目初始化流程

用户在飞书中发送：

- `开始创建项目`

系统按最简单可靠的方式收集两项信息：

1. 项目名称
2. Git 仓库地址或本地仓库路径

随后系统完成以下动作：

1. 创建本地项目记录
2. 记录管理员飞书 open_id
3. 创建项目群
4. 将管理员拉入群
5. 创建项目文档
6. 给管理员发送初始化结果
7. 在项目群发送欢迎消息

### 5.2 项目运行流程

项目创建完成后，系统基于项目上下文做以下事情：

1. 向项目群发送提醒
2. 向管理员单独通知风险
3. 生成日报或周报
4. 跟踪任务状态
5. 识别逾期、阻塞、长期无更新等异常


## 6. 必须保留的业务能力

### 6.1 项目域

必须有：

1. 项目创建
2. 项目查询
3. 项目与飞书群的绑定
4. 项目与飞书文档的绑定

建议数据字段：

- `id`
- `name`
- `git_repo_path`
- `feishu_group_id`
- `feishu_doc_url`
- `admin_open_id`
- `created_at`
- `updated_at`

### 6.2 成员域

必须有：

1. 管理员成员记录
2. 成员飞书 open_id 绑定
3. 项目内角色区分

建议最小字段：

- `id`
- `project_id`
- `name`
- `feishu_open_id`
- `role`

### 6.3 任务域

第一阶段不要求飞书任务双向全同步，但要保留扩展口。

必须支持：

1. 本地记录任务
2. 逾期提醒
3. 阻塞提醒
4. 周报汇总

### 6.4 消息域

必须支持：

1. 私聊用户发消息
2. 向群发送消息
3. 系统消息去重
4. 错误可追踪

### 6.5 飞书工作空间域

必须支持：

1. 创建群
2. 拉用户入群
3. 获取群分享链接
4. 创建项目文档
5. 向群和用户发送文本消息


## 7. 飞书集成策略

## 7.1 总体策略

飞书集成分成两层：

1. 输入层：Webhook 或消息事件接收用户输入
2. 执行层：优先使用 Feishu CLI 执行动作

原因：

1. Webhook 更适合接收事件
2. CLI 更适合做“创建群、发消息、建文档”这类明确动作
3. CLI 把执行能力和业务编排隔离开，更容易替换和排查

### 7.2 输入层要求

必须能稳定接收：

1. 私聊消息
2. 卡片按钮点击事件

但第一阶段建议：

1. 不把项目创建依赖在复杂表单卡片上
2. 优先使用文本引导式交互
3. 卡片只保留“开始创建项目”这类简单触发用途

### 7.3 执行层要求

通过 Feishu CLI 支持：

1. `im +messages-send`
2. `im chats create`
3. `im chat.members create`
4. `im chats link`
5. `docs +create`

后续可扩展：

1. `task`
2. `calendar`
3. `mail`
4. `wiki`
5. `drive`


## 8. 推荐的最小技术架构

建议保留三层，但一定要做薄：

### 8.1 接入层

负责：

1. 接收飞书 webhook
2. 验签
3. 解析事件
4. 转换成内部事件

不要在这一层直接写复杂业务逻辑。

### 8.2 业务层

负责：

1. 项目创建编排
2. 项目初始化编排
3. 消息路由
4. 通知规则
5. 报告生成触发

建议这里直接围绕明确用例写服务，不要过早泛化。

### 8.3 执行层

负责：

1. 调 CLI
2. 调数据库
3. 发事件
4. 记录日志

执行层必须满足：

1. 输入明确
2. 输出明确
3. 失败明确


## 9. 事件模型建议

建议只保留少数高价值事件：

1. `project.created`
2. `project.workspace_initialized`
3. `message.received`
4. `task.overdue`
5. `task.blocked`
6. `task.stale`
7. `schedule.daily_standup`
8. `schedule.weekly_report`

其中最重要的是：

- `project.created`

它必须成为后续所有初始化动作的统一触发点。


## 10. 第一阶段推荐实现顺序

### 阶段 1：最小可跑通

1. 启动服务
2. 接收飞书私聊消息
3. 响应 `开始创建项目`
4. 通过文本对话收集项目名和仓库地址
5. 本地建项目
6. 给用户返回“项目已创建”

### 阶段 2：飞书工作空间初始化

1. 触发 `project.created`
2. 创建项目群
3. 拉管理员进群
4. 创建项目文档
5. 给管理员发结果回执
6. 给项目群发欢迎消息

### 阶段 3：项目运行能力

1. 项目群通知
2. 任务异常提醒
3. 日报或周报生成
4. 管理员单独通知

### 阶段 4：增强能力

1. 飞书任务同步
2. 日历安排
3. 邮件通知
4. Drive 或 Wiki 资料沉淀
5. 更稳定的卡片交互


## 11. 这次重建必须规避的坑

### 11.1 不要把项目创建依赖在复杂卡片表单上

已踩过的问题：

1. 飞书卡片表单结构复杂
2. 事件体字段不稳定
3. 提交时容易出现参数错误或兼容问题
4. 表单一旦失败，用户感知非常差

结论：

第一阶段改用文本式分步收集，卡片只做简单触发。

### 11.2 不要把“动作成功”建立在伪成功返回上

已踩过的问题：

1. CLI 缺失时返回了伪成功
2. 业务层误以为动作已经执行
3. 用户看到“项目已创建”，但群、文档、后续流程都没发生

结论：

所有执行失败必须显式失败，不能假装成功。

### 11.3 不要把过多逻辑堆在 onboarding 或 webhook 中

已踩过的问题：

1. 接入层和业务层混在一起
2. 很难定位问题发生在哪一层
3. 修改一个入口会影响整条链路

结论：

项目初始化要抽成单独服务或动作。

### 11.4 不要过早追求全自动策略引擎

问题：

1. 规则、策略、LLM 三层并行，复杂度高
2. 初期最需要的是稳定，不是智能

结论：

MVP 先保留固定规则和明确流程，后面再考虑策略增强。

### 11.5 不要忽略日志与追踪

必须能回答这些问题：

1. Webhook 收到了没有
2. 消息解析到了没有
3. `project.created` 发出去了没有
4. 创建群成功没有
5. 拉人成功没有
6. 建文档成功没有
7. 回执发出去了没有


## 12. 可观测性要求

至少需要三类日志：

1. 接入日志
2. 业务编排日志
3. 动作执行日志

每个关键动作都必须打印：

- 输入摘要
- 成功或失败
- 错误原因
- 项目 ID
- 飞书用户 ID

推荐保留一个专门的飞书链路日志文件，例如：

- `feishu_trace.log`


## 13. 成功标准

只要满足以下条件，就说明第一阶段重建成功：

1. 用户在飞书里发送 `开始创建项目`
2. 系统通过文本方式拿到项目名和仓库地址
3. 本地数据库成功创建项目和管理员记录
4. 自动创建项目群
5. 管理员被拉入群
6. 自动创建项目文档
7. 管理员能收到初始化结果
8. 项目群里能看到欢迎消息
9. 任一步失败时，管理员能看到明确失败原因


## 14. 第二阶段成功标准

1. 已创建项目可以持续接收通知
2. 逾期和阻塞任务能触发提醒
3. 可以按项目生成日报或周报
4. 群通知和管理员通知都可用
5. 整条链路有日志可查


## 15. 推荐的重建边界

如果从头开始，建议只先保留以下模块：

- `app/api/webhooks.py`
- `app/feishu/client.py`
- `app/feishu/cli.py`
- `app/feishu/message_service.py`
- `app/feishu/onboarding.py`
- `app/actions/base.py`
- `app/actions/feishu_notify.py`
- `app/actions/feishu_workspace.py`
- `app/database.py`
- `app/models/schemas.py`

其余高级模块建议延后接回：

- LLM 决策
- 策略系统
- 策略演化
- 复杂调度
- 复杂卡片系统


## 16. 建议的重建产物

重建第一版时，最好交付以下明确产物：

1. 一张时序图
2. 一份数据库最小 schema
3. 一份飞书事件到内部事件的映射表
4. 一份 Feishu CLI 命令清单
5. 一份初始化链路的错误码与处理说明


## 17. 最终结论

新的 TeamFlow 不应该从“一个很复杂的智能代理系统”开始。

应该从“一个能在飞书里稳定完成项目初始化和项目通知闭环的小系统”开始。

先把下面这条链路做成铁三角：

`飞书输入 -> 项目创建 -> 飞书协作空间初始化`

然后再逐步往上叠加：

`任务提醒 -> 周报日报 -> 风险识别 -> 更强的 AI 能力`

如果这条主链路稳定，整个系统就有继续演进的价值。
