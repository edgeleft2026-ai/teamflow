# 平台支撑与可观测性模块 PRD

## 1. 模块目标

为 TeamFlow 提供稳定运行所需的平台能力，包括接入、执行、存储、调度、日志、审计和基础运维支持，确保产品在真实环境中可部署、可追踪、可定位问题。

## 2. 模块边界

### 2.1 本模块负责

1. 飞书事件接入。
2. 消息路由基础设施。
3. 内部事件总线。
4. 动作执行层封装。
5. 数据存储。
6. 定时调度。
7. 运行日志。
8. 审计记录。
9. 健康检查。
10. 配置管理。

### 2.2 本模块不负责

1. 具体业务文案。
2. AI 分析内容生成。
3. 产品侧复杂权限系统。
4. 多租户控制台。

## 3. 架构要求

建议采用三层：

1. 接入层：负责飞书事件、命令和卡片动作接入。
2. 业务编排层：负责状态机、事件分发和业务流程。
3. 执行层：负责调用飞书、数据库、模型和通知发送。

原则：

1. 接入层不承载复杂业务。
2. 编排层不直接操作外部系统。
3. 执行层返回统一结果。
4. 关键动作走标准事件、标准日志和审计记录。

## 4. 飞书接入要求

### 4.1 事件输入

必须支持：

1. Webhook 或 WebSocket 接入。
2. 私聊消息接收。
3. 群消息接收。
4. 简单卡片动作接收。

### 4.2 消息治理

必须支持：

1. 飞书事件验签或来源校验。
2. 事件去重。
3. Bot 自身消息过滤。
4. 异常事件记录。
5. 原始事件和内部事件关联。

## 5. 事件总线

内部事件总线至少支持：

1. 发布事件。
2. 消费事件。
3. 记录事件状态。
4. 基于幂等键忽略重复事件。
5. 失败事件可重试。

第一阶段核心事件：

1. `project.created`
2. `project.workspace_initialized`
3. `task.overdue`
4. `task.blocked`
5. `task.stale`
6. `schedule.daily_standup`
7. `schedule.weekly_report`

事件定义见 [数据与事件模型](../02-data-and-event-model.md)。

## 6. 执行层要求

执行层采用**双通道架构**：确定性通道（lark-cli subprocess）和智能通道（Agent + MCP）。

### 6.1 确定性通道

```text
Python 主进程
  → execution 模块构建 CLI 命令和参数
  → subprocess.run(["lark-cli", ...])
  → 解析 stdout JSON 输出（业务结果）
  → 捕获 stderr 日志
  → 返回 CLIResult(success, output, error, stderr_log)
```

适用场景：发消息、拉人入群、事件订阅等高频确定性动作。

### 6.2 智能通道

```text
Python 主进程
  → 编排层构建 AgentTask(description, context, complexity)
  → Agent Executor (LiteLLM tool-use 循环)
    → MCP Client → @larksuiteoapi/lark-mcp → 飞书 API
  → 返回 AgentResult(success, summary, actions, data)
```

适用场景：工作空间初始化、报告生成、风险分析等多步编排。

Agent 通过 MCP `tools/list` 动态发现可用工具，通过 `-t` 参数按里程碑启用工具集。

### 6.3 通道选择

编排层根据动作复杂度选择通道：参数完全确定的单步调用走确定性通道，多步编排或需要 AI 判断的走智能通道。默认确定性通道，降级安全。

### 6.4 确定性通道命令映射

| 业务动作 | CLI 命令 | 说明 |
|----------|----------|------|
| 发送消息 | `im +messages-send` | 支持 --chat-id / --user-id、--text / --markdown |
| 拉人入群 | `im +chat-members-add` | — |
| 获取群链接 | `im +chat-link` | — |
| 订阅事件 | `event +subscribe` | 长驻进程，--output-dir、--route |

### 6.5 统一结果格式

确定性通道返回 `CLIResult`：

```json
{
  "success": true,
  "output": {},
  "error": null,
  "stderr_log": "",
  "return_code": 0
}
```

智能通道返回 `AgentResult`：

```json
{
  "success": true,
  "summary": "已创建项目群和文档",
  "actions": [{"tool": "im.v1.chat.create", "args": {...}, "result": {...}}],
  "data": {"chat_id": "...", "doc_url": "..."},
  "error": null
}
```

### 6.5 事件订阅

事件订阅是唯一的长驻 CLI 进程：

```bash
lark-cli event +subscribe \
  --event-types im.message.receive_v1,... \
  --compact \
  --output-dir /tmp/teamflow/events \
  --route '^im\.message=dir:./events/im/'
```

主进程通过文件监听消费事件，支持进程重启不丢事件。

### 6.6 要求

1. 输入参数明确。
2. 输出结果结构化（JSON）。
3. 失败原因明确（CLI stderr 包含错误信息）。
4. 敏感信息脱敏（日志记录前脱敏）。
5. 确定性通道动作写入 ActionLog；智能通道工具调用链写入 Agent 审计日志。

## 7. 数据存储要求

第一阶段最小数据对象：

1. Project。
2. Member。
3. Task。
4. ConversationState。
5. EventLog。
6. ActionLog。
7. Observation。
8. Decision。

数据原则：

1. 关键业务对象必须可追溯。
2. 关键执行结果必须可回查。
3. 项目与飞书资源必须绑定。
4. 数据写入失败不得造成外部资源重复创建。

## 8. 调度要求

系统应支持定时触发：

1. 每日站会摘要。
2. 每周周报。
3. 逾期任务扫描。
4. 停滞任务扫描。
5. Git 活动扫描。
6. 项目健康检查。

调度要求：

1. 调度任务必须有幂等键。
2. 任务执行结果必须记录。
3. 任务失败必须可重试或可观察。
4. 多实例部署时不得重复执行同一调度任务。

## 9. 可观测性要求

### 9.1 日志类型

至少包含：

1. 接入日志。
2. 编排日志。
3. 执行日志。
4. 调度日志。
5. 模型调用日志摘要。

### 9.2 日志字段

关键日志需带上：

1. `request_id`
2. `event_id`
3. `project_id`
4. `actor_open_id`
5. `action_name`
6. `result`
7. `error_message`
8. `created_at`

### 9.3 健康检查

至少提供：

1. 轻量健康检查：服务是否存活。
2. 详细健康检查：数据库、飞书连接、调度器、模型配置状态。
3. 关键依赖状态输出。

## 10. 审计要求

下列动作必须具备审计记录：

1. 项目创建。
2. 项目空间初始化。
3. 高风险建议动作。
4. 用户审批操作。
5. 系统自动执行的重要动作。
6. 配置变更。

审计记录不得被业务流程覆盖或删除。

## 11. 配置要求

1. 飞书配置独立管理。
2. 模型配置独立管理。
3. 调度配置独立管理。
4. 环境变量和配置文件都可支持。
5. 配置缺失时必须在启动或健康检查中暴露。
6. 密钥不得进入普通日志。

## 12. 安全要求

1. 对外接口具备基础鉴权。
2. 飞书事件具备来源校验。
3. 日志和错误信息脱敏。
4. 高风险动作必须经过自治级别判断。
5. 用户权限校验必须可扩展。

## 13. 成功指标

1. 服务可用性。
2. 关键链路成功率。
3. 故障定位平均耗时。
4. 定时任务成功率。
5. 飞书消息接收与发送成功率。
6. 重复事件误处理次数。

建议第一阶段目标：

1. Mock 环境重复事件误处理次数为 0。
2. 核心动作 100% 有执行日志。
3. 健康检查 100% 覆盖数据库、飞书配置和调度器状态。

## 14. 验收标准

1. 飞书消息能够稳定进入系统。
2. 关键业务事件能够被正确分发。
3. 执行层动作结果明确可追踪。
4. 健康检查可以反映核心依赖状态。
5. 关键动作具备日志和审计记录。
6. 重复事件不会导致重复外部资源。
7. 日志中不出现密钥、令牌或 API Key。
