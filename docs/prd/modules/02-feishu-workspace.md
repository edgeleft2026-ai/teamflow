# 飞书工作空间模块 PRD

## 1. 模块目标

在项目创建成功后，自动为项目初始化飞书协作空间，包括项目群、项目文档、管理员回执和群欢迎消息，确保项目具备可立即使用的沟通和文档载体。

## 2. 模块边界

### 2.1 本模块负责

1. 消费 `project.created` 事件。
2. 创建或复用飞书项目群。
3. 拉项目管理员入群。
4. 获取群分享链接。
5. 创建项目文档。
6. 将管理员添加为文档协作者。
7. 将文档所有权转交给管理员。
8. 写回飞书资源绑定。
9. 向管理员发送初始化回执（卡片 + 文本降级）。
10. 在项目群发送欢迎消息。
11. 发布 `project.workspace_initialized` 事件。

### 2.2 本模块不负责

1. 项目创建引导。
2. 项目成员批量维护。
3. 项目文档持续更新。
4. 群权限精细化配置。
5. 项目运行期提醒和报告。

## 3. 触发方式

统一由 `project.created` 事件触发。模块不得自行扫描未初始化项目并偷偷执行，除非由平台调度显式触发补偿任务。

## 4. 输入输出

### 4.1 输入

| 字段 | 来源 | 必填 | 说明 |
|---|---|---|---|
| project_id | `project.created` | 是 | 项目 ID |
| project_name | `project.created` | 是 | 项目名称 |
| git_repo_path | `project.created` | 是 | 仓库地址或本地路径 |
| admin_open_id | `project.created` | 是 | 项目管理员 |

### 4.2 输出

1. `feishu_group_id`
2. `feishu_group_link`
3. `feishu_doc_url`
4. `workspace_status`
5. 每一步动作结果。
6. `project.workspace_initialized` 事件。

## 5. 主要流程

1. 接收 `project.created`。
2. 校验事件幂等键。
3. 读取项目基础信息。
4. 判断项目是否已有群或文档（workspace_status 检查）。
5. 创建或复用飞书项目群。
6. 拉管理员入群。
7. 获取群分享链接。
8. 创建或复用项目文档。
9. 将管理员添加为文档协作者。
10. 将文档所有权转交给管理员。
11. 写回项目记录。
12. 发送管理员回执（卡片进度 + 文本降级）。
13. 发送群欢迎消息。
14. 发布 `project.workspace_initialized`。

## 6. 功能定义

### 6.1 群创建

默认群名：

```text
TeamFlow | {项目名}
```

要求：

1. 创建前检查项目记录是否已有 `feishu_group_id`。
2. 创建成功后立即记录群 ID。
3. 群名过长时允许截断，但回执中应显示实际群名。

### 6.2 管理员入群

要求：

1. `admin_open_id` 有效时自动加入群。
2. 加入失败不影响后续文档创建。
3. 加入失败必须出现在管理员回执和执行日志中。

### 6.3 群链接获取

要求：

1. 能获取时写入项目记录。
2. 无权限或飞书不支持时，允许为空。
3. 为空时必须记录原因。

### 6.4 项目文档创建

项目文档至少包含：

1. 项目名称。
2. 项目负责人或管理员。
3. 仓库地址或本地路径。
4. 创建时间。
5. 后续状态区域。

创建前必须检查项目记录是否已有 `feishu_doc_url`。

### 6.5 文档权限管理

文档创建后需要：

1. 将管理员添加为文档协作者（`drive.v1.permission.add_collaborator`）。
2. 将文档所有权转交给管理员（`drive.v1.permission.transfer_owner`）。
3. 所有权转交失败时降级为仅添加协作者，不阻断流程。
4. 权限操作失败必须出现在管理员回执和执行日志中。

### 6.6 初始化回执

管理员回执必须包含：

1. 项目名称。
2. 群创建结果。
3. 管理员入群结果。
4. 群链接结果。
5. 文档创建结果。
6. 文档权限转交结果。
7. 失败步骤和失败原因。
8. 可重试建议。

回执优先使用卡片消息（`workspace_init_result_card`），支持分步进度更新。卡片更新失败时降级为纯文本回执。

### 6.7 群欢迎消息

群欢迎消息必须包含：

1. 当前群用途。
2. 当前项目名称。
3. 可用指令入口。
4. 项目文档链接，如果已创建成功。

## 7. 数据写回

项目记录需要写回：

1. `feishu_group_id`
2. `feishu_group_link`
3. `feishu_doc_url`
4. `workspace_status`
5. `updated_at`

执行动作需要写入 `ActionLog`，事件消费需要写入 `EventLog`。

## 8. 业务规则

1. 项目空间初始化必须幂等。
2. 同一项目已有群或文档时，不重复创建。
3. 每一步动作必须返回明确成功或失败。
4. 单步失败不应遮蔽其他已完成结果。
5. 回执消息必须包含失败原因。
6. 外部资源创建成功后不得因后续步骤失败而自动删除。

## 9. 执行策略

工作空间初始化采用双通道执行：

- **Agent 智能通道**：编排层构建任务描述（项目名、管理员 ID、初始化清单），委托 Agent 通过 ToolProvider 调用飞书 API。Agent 负责多步编排（创建群 → 拉人 → 创建文档 → 添加协作者 → 转交所有权 → 发欢迎语）和部分失败处理。
- **确定性通道**：初始化结果回写数据库、发送回执消息走确定性通道（lark-oapi SDK）。

Agent 工具集：`im.v1.chat.create`, `im.v1.chat.members.create`, `im.v1.message.create`, `docx.v1.document.create`, `drive.v1.permission.add_collaborator`, `drive.v1.permission.transfer_owner`。

编排层保留降级路径：Agent 执行超时或失败时，退回分步确定性通道执行。

进度追踪：每步完成后通过 `update_card_message` 更新同一张进度卡片，用户可实时看到初始化进度。卡片更新失败时降级为纯文本回执。

统一执行结果格式至少包含：

1. `success`
2. `action_name`
3. `target`
4. `output`
5. `error_message`

## 10. 异常处理

| 异常 | 处理要求 |
|---|---|
| ToolProvider 工具调用失败 | 降级为确定性通道分步执行 |
| 群创建失败 | 记录失败，继续判断是否能发送管理员回执 |
| 管理员入群失败 | 记录失败，继续获取链接和创建文档 |
| 群链接获取失败 | 允许继续，写入失败原因 |
| 文档创建失败 | 保留群结果，回执中说明 |
| 文档权限转交失败 | 降级为添加协作者，不阻断流程 |
| 数据写回失败 | 记录可重试错误，不重复创建外部资源 |
| 回执发送失败 | 写入日志和动作记录，支持后续补偿 |
| 卡片更新失败 | 降级为纯文本回执，记录日志 |

## 11. 成功指标

1. 空间初始化成功率。
2. 单步动作成功率。
3. 初始化平均耗时。
4. 重复事件导致重复资源创建次数。
5. 管理员回执发送成功率。

建议第一阶段目标：

1. Mock 环境重复事件重复资源创建次数为 0。
2. 正常配置下空间初始化成功率达到 95%。
3. 单步失败时回执覆盖率达到 100%。

## 12. 验收标准

1. 触发 `project.created` 后能自动进入空间初始化流程。
2. 能成功创建或复用项目群。
3. 能把管理员拉入群，失败时可见原因。
4. 能创建或复用项目文档。
5. 能将群和文档信息写回项目记录。
6. 管理员能收到初始化结果回执。
7. 项目群能收到欢迎消息。
8. 重复消费同一事件不会重复创建群或文档。
