# 项目入口与初始化模块 PRD

## 1. 模块目标

为用户提供一个简单、稳定、低学习成本的项目创建入口，让用户可以直接在飞书中发起项目，并完成项目基础信息收集、项目记录创建、管理员绑定和 `project.created` 事件发布。

## 2. 模块边界

### 2.1 本模块负责

1. 欢迎引导和帮助入口。
2. "开始创建项目"触发识别。
3. 文本式分步信息收集。
4. 卡片表单式信息收集。
5. 用户 open_id 识别。
6. 创建流程状态持久化。
7. 项目记录落库。
8. 项目管理员绑定。
9. 创建结果回执（文本 + 卡片进度）。
10. `project.created` 事件发布。

### 2.2 本模块不负责

1. 创建飞书群。
2. 创建项目文档。
3. 拉成员入群。
4. 项目运行期风险提醒。
5. 多人共同编辑同一创建流程。

下游空间初始化由 [飞书工作空间模块](./02-feishu-workspace.md) 负责。

## 3. 核心用户

1. 项目发起人。
2. 项目管理员。
3. 项目负责人。

第一阶段中，项目发起人默认成为项目管理员。

## 4. 用户场景

### 4.1 新项目启动

用户在飞书中私聊 TeamFlow，发送“开始创建项目”，系统开始引导创建。

### 4.2 创建流程恢复

用户在创建过程中中断，稍后再次发送消息，系统能识别当前阶段并继续引导或允许重新开始。

### 4.3 创建失败反馈

项目创建失败时，用户需要知道失败步骤、原因和下一步可执行动作。

## 5. 输入输出

### 5.1 输入

| 字段 | 来源 | 必填 | 校验 |
|---|---|---|---|
| project_name | 用户文本输入 / 卡片表单 | 是 | 去除首尾空白后非空 |
| git_repo_path | 用户文本输入 / 卡片表单 | 是 | 去除首尾空白后非空 |
| admin_open_id | 飞书消息上下文 | 是 | 必须能从消息中解析 |
| chat_id | 飞书消息上下文 | 是 | 用于回复和会话状态 |
| request_id | 卡片回调上下文 | 否 | 卡片表单提交时必填，用于去重 |
| open_message_id | 卡片回调上下文 | 否 | 卡片表单提交时必填，用于更新进度卡片 |

### 5.2 输出

1. 项目 ID。
2. 项目名称。
3. 项目创建状态。
4. 成功或失败回执。
5. `project.created` 事件。

## 6. 交互流程

### 6.1 欢迎消息

触发条件：

1. 用户首次私聊 Bot。
2. 用户发送 `/help`。
3. 用户输入无法识别且当前无活跃流程。

消息应包含：

1. 可创建项目。
2. 可查看帮助。
3. 可查看项目状态。

### 6.2 创建引导

推荐流程：

1. 用户发送“开始创建项目”。
2. 系统询问项目名称。
3. 用户输入项目名称。
4. 系统询问 Git 仓库地址或本地路径。
5. 用户输入仓库信息。
6. 系统展示确认信息并开始创建。
7. 系统返回创建结果。

### 6.3 卡片表单创建

推荐流程：

1. 用户发送"开始创建项目"。
2. 系统发送项目创建表单卡片（`project_create_form_card`）。
3. 用户在卡片中填写项目名称和 Git 仓库地址，点击"提交"。
4. 系统接收卡片回调（`card.action.trigger`），记录 `ProjectFormSubmission`。
5. 系统立即返回 toast 确认，后台 Worker 开始执行创建流程。
6. 创建过程中，系统逐步更新同一张卡片显示当前步骤和状态。
7. 创建完成后，卡片显示最终结果（成功/失败）。

优势：

1. 一次交互完成信息收集，无需多轮对话。
2. 进度卡片实时更新，用户无需等待。
3. 表单字段校验即时反馈（toast 提示）。

### 6.4 状态流转

创建项目状态机：

```text
idle
  -> collecting_project_name
  -> collecting_repo
  -> creating_project
  -> created
```

异常状态：

```text
creating_project -> failed
collecting_project_name -> idle
collecting_repo -> idle
```

## 7. 数据要求

本模块至少写入：

1. `Project`
2. `ConversationState`
3. `ProjectFormSubmission`
4. `EventLog`
5. `ActionLog`

字段定义见 [数据与事件模型](../02-data-and-event-model.md)。

## 8. 业务规则

1. 项目名称不能为空。
2. 仓库地址或本地路径不能为空。
3. 项目管理员必须来自当前触发用户的 open_id。
4. 创建成功后必须发布 `project.created`。
5. 输入层不直接调用飞书群或文档创建动作。
6. 用户重复触发创建时，必须明确提示覆盖当前流程或重新开始。
7. 创建流程状态必须有过期时间，避免长期占用会话。

## 9. 异常处理

| 异常 | 处理要求 |
|---|---|
| 用户输入为空 | 提示重新输入，不进入下一步 |
| 用户中途退出 | 保留会话状态，并提示当前阶段 |
| 用户重复触发创建 | 允许重新开始，旧流程标记为中止或覆盖 |
| open_id 解析失败 | 明确提示身份识别失败，并记录日志 |
| 项目落库失败 | 返回失败步骤和原因，不发布 `project.created` |
| 事件发布失败 | 项目保持 created 状态，并记录可重试事件 |
| 卡片表单字段为空 | 返回 toast 错误提示，不进入创建流程 |
| 卡片表单重复提交 | 返回当前状态卡片，不重复创建项目 |
| 卡片更新失败 | 降级为纯文本回执，记录日志 |

## 10. 成功指标

1. 项目创建成功率。
2. 创建流程完成率。
3. 用户从触发到创建完成的平均耗时。
4. 由于输入错误导致的失败率。
5. `project.created` 事件发布成功率。

建议第一阶段目标：

1. Mock 环境项目创建成功率达到 95%。
2. 正常输入下创建流程不超过 5 个用户交互轮次。
3. 创建失败必须 100% 返回用户可读错误。

## 11. 验收标准

1. 用户发送"开始创建项目"后系统能进入创建流程。
2. 系统能依次收集项目名称和仓库信息。
3. 空输入不会创建项目。
4. 项目记录能成功写入数据库。
5. 系统能返回项目 ID 和项目名。
6. 系统能发布带幂等键的 `project.created` 事件。
7. 创建失败时用户能看到失败步骤、原因和下一步建议。
8. 卡片表单能正确展示项目创建表单。
9. 卡片表单提交后能实时更新进度卡片。
10. 重复表单提交不会创建重复项目。
