# TeamFlow 测试策略

## 1. 文档目的

本文定义 TeamFlow 第一阶段的测试分层、测试数据和真实环境验证方式。目标不是追求一次性覆盖所有场景，而是确保主链路稳定、问题可定位、失败可恢复。

## 2. 测试分层

### 2.1 单元测试

覆盖纯业务逻辑：

1. 创建流程状态机。
2. 输入校验。
3. 事件 payload 构造。
4. 幂等键生成。
5. 消息路由规则。
6. 提醒去重和聚合。
7. 自治级别判断。

### 2.2 集成测试

覆盖模块之间的协作：

1. 飞书事件解析到内部消息对象。
2. 项目创建流程到数据库写入。
3. `project.created` 到空间初始化编排。
4. 执行层动作结果写回项目记录。
5. 定时任务生成提醒和报告。
6. AI 输出失败后的降级流程。

### 2.3 契约测试

覆盖外部接口和执行层封装：

1. Feishu CLI 调用输入输出格式。
2. 飞书事件 payload 解析。
3. 数据库读写接口。
4. 模型调用接口。
5. 健康检查响应格式。

### 2.4 端到端测试

覆盖真实用户路径：

1. 用户在飞书私聊中发送“开始创建项目”。
2. 系统完成信息收集。
3. 系统创建项目记录。
4. 系统初始化项目群和文档。
5. 管理员收到回执。
6. 项目群收到欢迎消息。
7. 用户执行 `/status` 查询项目状态。

## 3. 推荐测试数据

### 3.1 项目

1. 正常项目：`TeamFlow Demo`
2. 长名称项目：用于验证群名和文档标题。
3. 重复名称项目：用于验证提示和幂等。

### 3.2 用户

1. 有效管理员 open_id。
2. 无效或缺失 open_id。
3. 群内普通成员 open_id。

### 3.3 任务

1. 已逾期未完成任务。
2. 阻塞任务。
3. 长时间无更新任务。
4. 已完成任务。
5. 无负责人任务。

## 4. Mock 环境验证

在没有真实飞书租户时，应通过 mock 完成以下验证：

1. 模拟私聊消息事件。
2. 模拟群聊 @Bot 事件。
3. 模拟重复事件投递。
4. 模拟飞书群创建成功。
5. 模拟飞书群创建失败。
6. 模拟文档创建成功。
7. 模拟文档创建失败。
8. 模拟消息发送失败。

Mock 测试必须断言：

1. 数据库状态变化正确。
2. 事件被正确发布或忽略。
3. 错误被写入日志。
4. 用户可见回执内容包含失败原因。

## 5. 真实飞书环境验证

### 5.1 前置条件

1. 可用的飞书企业账号。
2. 已创建飞书应用。
3. 已配置 App ID、App Secret 和事件订阅。
4. 已申请必要权限。
5. 已准备测试群或测试用户。

### 5.2 必测路径

1. 私聊 Bot，发送 `/help`。
2. 私聊 Bot，发送“开始创建项目”。
3. 输入项目名称。
4. 输入仓库地址或本地路径。
5. 检查项目记录是否落库。
6. 检查飞书项目群是否创建。
7. 检查管理员是否入群。
8. 检查项目文档是否创建。
9. 检查管理员回执。
10. 检查群欢迎消息。
11. 在群内执行 `/status`。

### 5.3 真实环境失败场景

1. App Secret 配置错误。
2. 缺少建群权限。
3. 缺少文档创建权限。
4. 管理员 open_id 无效。
5. 飞书接口限流。
6. 回调重复投递。

每个失败场景都应验证：

1. 用户能看到可读错误。
2. 日志能定位失败动作。
3. 已成功步骤不被回滚破坏。
4. 重试不会制造重复资源。

## 6. 回归测试清单

每次改动以下区域后，必须跑主链路回归：

1. 飞书事件解析。
2. 会话状态机。
3. 项目表结构。
4. 事件总线。
5. Feishu CLI 封装。
6. 消息发送封装。
7. 定时调度。
8. AI 调用和提示词模板。

## 7. 最低发布门槛

发布前至少满足：

1. 单元测试覆盖创建流程、幂等和消息路由核心逻辑。
2. 集成测试覆盖项目创建到空间初始化。
3. Mock 环境通过重复事件和单步失败测试。
4. 真实飞书环境完成一次端到端创建。
5. 日志中没有密钥泄露。
6. [验收清单](./03-acceptance-checklist.md) 中的阻断项全部通过。
