# TeamFlow 产品总纲 PRD

## 1. 产品名称

TeamFlow

## 2. 产品定位

TeamFlow 是一个运行在飞书场景中的 AI 项目协作助手，帮助团队完成项目创建、协作空间初始化、项目状态同步、风险提醒和周期性汇报。

TeamFlow 不是通用聊天机器人，也不是完整企业流程平台。第一阶段的核心目标是成为一个稳定、可追踪、可逐步扩展的项目协作执行器。

## 3. 产品愿景

让团队在飞书里直接发起和管理项目，让系统自动完成基础协作设施搭建，并持续基于项目数据提供提醒、分析和建议。

## 4. 目标用户与角色

### 4.1 统一角色

1. 项目发起人：在飞书中发起项目创建流程的人。
2. 项目管理员：创建流程绑定的唯一管理者，默认接收初始化回执和重点风险提醒。
3. 项目负责人：对项目交付负责的人，第一阶段可与项目管理员为同一人。
4. 项目成员：在项目群中接收通知、查看任务和参与协作的人。
5. 团队管理者：关注多个项目状态和风险的人，第一阶段不单独建设管理后台。

### 4.2 第一阶段角色约束

1. 每个项目必须绑定一个项目管理员。
2. 项目管理员默认具备项目创建、初始化结果查看和高风险动作审批权限。
3. 项目成员权限先保持轻量，只支持接收群消息和发起基础查询。
4. 多项目管理视图暂不纳入第一阶段。

## 5. 目标问题

当前项目协作常见问题包括：

1. 新项目创建后，建群、建文档、通知成员等动作分散且重复。
2. 项目、负责人、飞书群和项目文档缺少统一绑定。
3. 任务逾期、阻塞、停滞和成员负载异常发现较晚。
4. 站会、周报和风险材料依赖人工整理，成本高且不稳定。
5. 管理动作分散在多个工具中，缺少可追踪闭环。

## 6. 核心价值

1. 快速起盘：在飞书中发起项目后，自动完成项目记录和协作空间初始化。
2. 持续协同：围绕项目群和项目管理员进行通知、提醒和同步。
3. 风险前置：对逾期、阻塞、长期无进展、成员过载等情况提前提醒。
4. 自动整理：基于项目数据生成站会摘要、周报和风险分析。
5. 可追踪执行：关键动作都有输入、输出、失败原因、日志和审计记录。

## 7. 第一阶段范围

第一阶段采用 M0 到 M4 分步交付，完整边界见 [MVP 范围](./01-mvp-scope.md)。

### 7.1 必须覆盖

1. 飞书消息接入和消息发送。
2. 项目创建入口和文本式分步引导。
3. 项目记录、成员身份和会话状态落库。
4. 飞书项目群创建、管理员入群、群链接获取和项目文档创建。
5. 初始化结果回执和项目群欢迎消息。
6. 逾期、阻塞、停滞和成员负载异常提醒。
7. 每日站会摘要、每周周报和基础风险分析。
8. `/help`、`/status`、`/tasks`、`/risk`、`/standup`、`/report` 基础查询。
9. 日志、审计、健康检查和关键链路可观测性。

### 7.2 暂不覆盖

1. 复杂多步卡片表单。
2. 高自由度工作流引擎。
3. 多平台消息接入。
4. 完整企业审批平台。
5. 多项目管理后台。
6. 重型知识库管理。
7. 复杂权限系统。
8. 自动删除数据、移除成员、关闭项目等高风险动作。

## 8. 产品原则

1. 稳定优先：先保证主链路可用，再提升交互复杂度。
2. 闭环优先：每个核心动作都必须有明确输入、执行结果和反馈。
3. 飞书优先：核心体验聚焦飞书，不分散入口。
4. 数据优先：分析、提醒和建议必须基于项目数据。
5. 可追踪优先：关键动作必须可定位、可审计、可解释。
6. 渐进智能：先做规则化和结构化能力，再叠加更强 AI 能力。
7. 人可接管：高风险动作必须支持人工确认或拒绝。

## 9. 核心业务闭环

### 9.1 项目创建闭环

1. 用户在飞书中发起项目创建。
2. 系统收集项目名称和仓库信息。
3. 系统识别用户 open_id 并绑定项目管理员。
4. 系统创建项目记录并持久化创建流程状态。
5. 系统发布 `project.created` 事件。
6. 系统返回创建结果。

### 9.2 协作空间初始化闭环

1. 系统消费 `project.created` 事件。
2. 系统创建或复用飞书项目群。
3. 系统拉项目管理员入群。
4. 系统获取群链接并创建项目文档。
5. 系统写回群 ID、群链接、文档 URL 和初始化状态。
6. 系统发送管理员回执和群欢迎消息。
7. 系统发布 `project.workspace_initialized` 事件。

### 9.3 项目运行闭环

1. 系统定时或按事件采集项目状态、任务状态和 Git 活动。
2. 系统通过规则、策略或 AI 分析识别异常和风险。
3. 系统按消息路由规则输出群提醒或私聊提醒。
4. 系统生成站会摘要、周报和风险分析。
5. 用户继续在飞书中查询、确认或拒绝后续动作。

## 10. 产品模块划分

1. 项目入口与初始化模块。
2. 飞书工作空间模块。
3. 项目运行与协作模块。
4. AI 分析与决策模块。
5. 平台支撑与可观测性模块。

## 11. 核心对象

详细字段、状态和事件 payload 见 [数据与事件模型](./02-data-and-event-model.md)。

1. 项目：承载项目基础信息、管理员、状态和飞书资源绑定。
2. 成员：承载项目成员身份、飞书 open_id 和角色关系。
3. 任务：承载工作项、负责人、状态、截止时间和风险状态。
4. 会话状态：承载用户当前创建流程和上下文。
5. 事件：承载系统内部业务信号和幂等控制。
6. 观察记录：承载风险扫描、站会、周报和分析结果。
7. 决策记录：承载自动动作、审批动作和 AI 建议。

## 12. 关键功能清单

### 12.1 输入能力

1. 私聊消息触发。
2. 群消息触发。
3. 简单卡片按钮触发。
4. 指令式输入。
5. 文本式引导输入。

### 12.2 执行能力

执行层基于 teamflow-cli（从飞书官方 lark-cli 编译的 Go 二进制），Python 主进程通过 subprocess 调用。

1. 创建项目记录。
2. 创建飞书群（`im +chat-create`）。
3. 拉管理员入群（`im +chat-members-add`）。
4. 获取群分享链接（`im +chat-link`）。
5. 创建项目文档（`docs +create`）。
6. 发送群消息（`im +messages-send`）。
7. 发送私聊消息（`im +messages-send`）。
8. 记录日志和审计（Transport Extension 自动记录）。

### 12.3 分析能力

1. 项目状态汇总。
2. 逾期提醒。
3. 阻塞提醒。
4. 停滞提醒。
5. 成员负载分析。
6. 站会摘要。
7. 周报生成。
8. 风险分析。

## 13. 非功能要求

### 13.1 稳定性

1. 核心链路必须可重试。
2. 执行失败必须显式反馈。
3. 输入层与执行层必须解耦。
4. 已成功的外部资源不得因后续步骤失败而被误删。

### 13.2 可观测性

1. 可查询消息接入状态。
2. 可查询项目初始化执行结果。
3. 可查询关键事件是否触发。
4. 可查询飞书动作执行结果。
5. 关键日志必须能关联项目 ID、事件 ID 和动作名称。

### 13.3 安全性

1. 对外接口需具备基础鉴权。
2. 飞书事件需校验来源。
3. 高风险动作需审批。
4. 用户与项目权限关系需可扩展。
5. API Key、App Secret 和模型密钥不得进入日志。

## 14. 里程碑规划

### M0：飞书交互链路打通

完成飞书事件接入、私聊和群消息接收、主动发送消息、身份识别、消息去重和健康检查。

### M1：项目创建可用

完成项目创建引导、项目落库、管理员绑定、会话状态持久化、`project.created` 事件和创建结果回执。

### M2：飞书工作空间可用

完成群创建、管理员入群、群链接获取、文档创建、初始化写回、回执、欢迎消息和 `project.workspace_initialized` 事件。

### M3：项目运行协作可用

完成风险提醒、站会摘要、周报、风险分析、指令式查询、消息路由和去重聚合。

### M4：AI 能力增强

完成规则、策略、AI 分析分层决策，模型层级路由，自治级别控制和 AI 输出降级。

## 15. 第一阶段成功标准

1. 用户能在飞书中发起项目创建，并在失败时获得可执行反馈。
2. 项目创建成功后能自动初始化飞书项目群和项目文档。
3. 管理员能收到包含每步结果的初始化回执。
4. 项目群能收到可读欢迎消息和后续提醒。
5. 系统能基于项目数据输出基础风险提醒、站会摘要和周报。
6. 关键动作具备日志、审计和健康检查支撑。
7. 第一阶段验收清单全部通过，详见 [验收清单](./03-acceptance-checklist.md)。
