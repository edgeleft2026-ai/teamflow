# TeamFlow 验收清单

## 1. 文档目的

本文用于在开发完成后逐项验收 TeamFlow 第一阶段能力。每条验收项都应能通过人工操作、自动化测试或日志证据验证。

## 2. M0 飞书交互验收

- [ ] 私聊消息能进入系统并被记录为接入日志。
- [ ] 群消息只有在 @Bot 或明确命令时触发处理。
- [ ] Bot 自己发出的消息不会再次触发业务处理。
- [ ] 同一个飞书事件重复投递时只处理一次。
- [ ] 系统能主动发送私聊消息。
- [ ] 系统能主动发送群消息。
- [ ] 消息上下文能提取用户 open_id。
- [ ] `/health` 返回服务存活状态。
- [ ] 飞书配置缺失时系统能给出明确错误，而不是静默失败。

## 3. M1 项目创建验收

- [ ] 用户发送“开始创建项目”后进入创建流程。
- [ ] 用户首次触达或发送 `/help` 时能看到欢迎引导。
- [ ] 系统能依次收集项目名称和仓库信息。
- [ ] 项目名称为空时提示重新输入，不创建项目。
- [ ] 仓库地址或本地路径为空时提示重新输入，不创建项目。
- [ ] 创建流程状态能持久化，服务重启后不会造成不可解释状态。
- [ ] 重复发送“开始创建项目”时系统允许重新开始或明确提示当前流程。
- [ ] 项目记录成功写入数据库。
- [ ] 项目记录包含项目 ID、名称、仓库信息、管理员 open_id、状态和时间戳。
- [ ] 创建成功后系统发送包含项目 ID 和项目名称的回执。
- [ ] 创建失败后系统发送失败步骤、失败原因和下一步建议。
- [ ] 创建成功后发布 `project.created` 事件。

## 4. M2 飞书工作空间验收

- [ ] `project.created` 事件能触发空间初始化流程。
- [ ] 系统能创建格式为 `TeamFlow | {项目名}` 的项目群。
- [ ] 系统能将项目管理员加入项目群。
- [ ] 系统能获取或记录群链接；无法获取时要说明原因。
- [ ] 系统能创建项目文档。
- [ ] 项目文档至少包含项目名称、负责人、仓库信息和创建时间。
- [ ] 群 ID、群链接、文档 URL 和初始化状态能写回项目记录。
- [ ] 管理员能收到包含每一步成功或失败状态的回执。
- [ ] 项目群能收到欢迎消息。
- [ ] 重复消费同一项目创建事件不会重复创建群或文档。
- [ ] 群创建成功但文档创建失败时，已创建群不会被误删，回执中能看到部分失败。
- [ ] 初始化结束后发布 `project.workspace_initialized` 事件。

## 5. M3 项目运行与协作验收

- [ ] 系统能识别已过截止时间且未完成的任务，并生成逾期提醒。
- [ ] 系统能识别阻塞任务，并生成阻塞提醒。
- [ ] 系统能识别长期无进展任务或分支，并生成停滞提醒。
- [ ] 系统能识别成员负载明显异常，并生成负载提醒。
- [ ] 同一提醒在配置时间窗口内不会重复刷屏。
- [ ] 多条同类提醒可聚合成一条可读消息。
- [ ] 每日站会摘要包含昨日完成、进行中、需关注、健康度和今日建议。
- [ ] 每周周报包含本周概要、关键成果、进行中、风险问题、下周计划和贡献统计。
- [ ] 风险分析能区分进度风险、人员风险和技术风险。
- [ ] 数据不足时，报告明确说明缺少哪些数据。
- [ ] `/status` 能返回项目当前状态。
- [ ] `/tasks` 能返回任务列表或空状态说明。
- [ ] `/risk` 能返回当前风险摘要。
- [ ] `/standup` 能生成或查询站会摘要。
- [ ] `/report` 能生成或查询周报。
- [ ] 群通知和管理员私聊通知符合消息路由规则。

## 6. M4 AI 能力验收

- [ ] 逾期、停滞、定时汇报等确定性场景优先由规则层处理。
- [ ] 成员过载、通知策略等场景可由策略层处理。
- [ ] 复杂风险归因和自然语言问答可进入 AI 分析层。
- [ ] fast、smart、reasoning 模型层级能按任务类型路由。
- [ ] AI 输出区分事实、判断和建议。
- [ ] AI 输出引用或概括项目数据依据。
- [ ] 无数据依据时 AI 不强行给结论。
- [ ] AI 调用失败时系统能退回规则或基础文案。
- [ ] `auto` 动作可以直接执行并记录。
- [ ] `approval` 动作必须等待人工确认。
- [ ] `forbidden` 动作不得自动执行。

## 7. 平台与可观测性验收

- [ ] 接入日志记录消息进入系统的时间、来源和处理结果。
- [ ] 编排日志记录事件分发、业务流程和状态变化。
- [ ] 执行日志记录外部动作输入摘要、结果和失败原因。
- [ ] 审计记录覆盖项目创建、空间初始化、高风险建议、用户审批和系统自动执行的重要动作。
- [ ] `/health` 能反映服务存活。
- [ ] 详细健康检查能反映飞书连接、数据库和调度器状态。
- [ ] 关键动作日志能通过项目 ID 串联排查。
- [ ] 密钥、令牌和 API Key 不出现在日志中。
- [ ] 定时任务失败可被记录并重试或告警。

## 8. 发布前阻断项

以下任一问题未解决，不建议发布：

1. 飞书消息收发链路不可用。
2. 项目创建成功但无法落库。
3. 重复事件会创建重复群或重复文档。
4. 初始化失败没有任何用户可见反馈。
5. 日志中出现密钥或访问令牌。
6. 高风险动作绕过审批直接执行。
7. 健康检查无法反映核心依赖状态。
