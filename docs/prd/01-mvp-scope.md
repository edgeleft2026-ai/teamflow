# TeamFlow MVP 范围

## 1. 文档目的

本文用于明确 TeamFlow 第一阶段的交付边界。它回答三个问题：

1. 哪些能力必须做。
2. 哪些能力明确不做。
3. 每个里程碑完成到什么程度才算可验收。

## 2. MVP 总目标

MVP 只追求一条稳定主链路：

```text
用户在飞书中发起项目
  -> 系统完成项目创建
  -> 系统初始化飞书协作空间
  -> 系统可以围绕该项目提醒、汇报和查询
```

MVP 不追求复杂自动化，也不追求完整企业项目管理能力。所有能力都必须服务于“项目协作闭环可用”。

## 3. M0：飞书交互链路打通

### 3.1 目标

验证飞书消息能稳定进入系统，系统能向用户和群发送消息。

### 3.2 必须交付

1. 飞书应用配置读取。
2. Webhook 或 WebSocket 事件接入。
3. 私聊消息接收与解析。
4. 群消息接收与解析，群内仅响应明确 @Bot 或命令。
5. Bot 自身消息过滤。
6. 消息事件去重。
7. 主动发送私聊消息。
8. 主动发送群消息。
9. 从消息上下文提取用户 open_id。
10. `/health` 轻量健康检查。
11. 一条端到端验证链路：用户发消息，系统收到，系统回复。

### 3.3 不做

1. 复杂卡片表单。
2. 多平台消息接入。
3. 多租户管理后台。

### 3.4 验收口径

1. 同一个飞书事件重复投递时只处理一次。
2. 私聊和群聊都能完成收发验证。
3. 机器人不会响应自己发出的消息。
4. 健康检查能返回服务存活状态。

## 4. M1：项目创建可用

### 4.1 目标

用户能在飞书中完成项目创建，系统能落库并发布项目创建事件。

### 4.2 必须交付

1. 欢迎引导和 `/help`。
2. “开始创建项目”触发识别。
3. 文本式分步引导。
4. 会话状态持久化。
5. 项目名称收集与校验。
6. Git 仓库地址或本地路径收集与校验。
7. 项目记录写入数据库。
8. 项目管理员 open_id 绑定。
9. `project.created` 事件发布。
10. 成功回执和失败回执。

### 4.3 不做

1. 一次性收集大量项目字段。
2. 多人共同编辑同一个创建流程。
3. 自动识别仓库内任务和里程碑。

### 4.4 验收口径

1. 创建流程可以从中断状态恢复或重新开始。
2. 项目名称和仓库信息为空时不会落库。
3. 创建成功后必须能查到项目记录。
4. `project.created` 必须带有项目 ID、项目名称、管理员 open_id 和幂等键。

## 5. M2：飞书工作空间可用

### 5.1 目标

项目创建后自动初始化飞书项目群和项目文档，并向管理员回报每一步结果。

### 5.2 必须交付

1. 监听 `project.created`。
2. 创建或复用飞书项目群。
3. 拉项目管理员入群。
4. 获取群分享链接。
5. 创建项目文档。
6. 写回群 ID、群链接、文档 URL 和初始化状态。
7. 向管理员发送初始化结果回执。
8. 在项目群发送欢迎消息。
9. 发布 `project.workspace_initialized`。
10. 幂等保护，避免重复创建群和文档。

### 5.3 不做

1. 复杂群成员批量维护。
2. 项目文档的复杂模板系统。
3. 文档持续自动更新。

### 5.4 验收口径

1. 重复消费同一个 `project.created` 不会重复创建外部资源。
2. 单步失败时保留已成功结果，并在回执中说明失败步骤。
3. 初始化完成后项目记录必须能查到飞书资源绑定。

## 6. M3：项目运行与协作可用

### 6.1 目标

项目进入运行期后，系统能围绕任务、Git 活动和项目状态输出基础提醒、汇报和查询结果。

### 6.2 必须交付

1. 逾期任务提醒。
2. 阻塞任务提醒。
3. 停滞任务提醒。
4. 成员负载异常提醒。
5. 每日站会摘要。
6. 每周周报。
7. 基础风险分析。
8. `/status`、`/tasks`、`/risk`、`/standup`、`/report`。
9. 群通知和管理员私聊通知路由。
10. 提醒去重和聚合。
11. 定时扫描任务。

### 6.3 不做

1. 飞书任务双向同步。
2. 日历联动。
3. 邮件通知。
4. 跨项目仪表盘。

### 6.4 验收口径

1. 同一风险短时间内不会重复刷屏。
2. 报告内容必须基于已有数据，数据不足时明确说明。
3. 查询命令必须返回项目上下文内的明确信息或空状态说明。

## 7. M4：AI 能力增强

### 7.1 目标

在规则和策略基础上叠加 AI 分析能力，让风险判断、报告总结和自然语言问答更有用，但不让 AI 直接执行高风险动作。

### 7.2 必须交付

1. 规则层、策略层、AI 分析层的分层命中机制。
2. fast、smart、reasoning 三类模型能力路由。
3. auto、approval、forbidden 三类自治级别。
4. 风险分析提示词。
5. 站会摘要提示词。
6. 周报提示词。
7. 任务建议提示词。
8. AI 输出失败降级到规则或基础文案。

### 7.3 不做

1. 自动删除数据。
2. 自动移除成员。
3. 自动关闭项目。
4. 自动改写项目策略并立即生效。

### 7.4 验收口径

1. AI 输出必须区分事实、判断和建议。
2. 数据不足时不得编造结论。
3. 需要人工确认的动作必须进入审批状态，而不是直接执行。

## 8. MVP 之外的候选能力

1. 多项目管理视图。
2. 飞书任务双向同步。
3. 日历和会议联动。
4. 邮件和其他 IM 平台通知。
5. 复杂审批模板。
6. 项目知识库自动维护。
7. 多租户后台。
